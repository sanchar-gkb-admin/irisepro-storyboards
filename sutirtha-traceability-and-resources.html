Why persist the code (your call — "definitely persistent and deterministic")
  • Same value, now stored. Today the code is a deterministic HMAC(student id, APP_KEY) → 3F9C1. We backfill every existing student with the exact same value, so no evaluator's "Candidate 3F9C1" reference ever shifts.
  • Rotation-proof. Once stored, an APP_KEY rotation can never silently change a student's code (today it would, breaking old references).
  • A real join key. Stored → indexable/unique → the traceability query is a clean join, and the code can appear on scorecards, exports, and dispute tickets.
  • Minted at enrolment (and defensively at student creation) via a model hook, so every student always has one.
#11

External resource access — Learning Material in a task

The person who matters here is a student mid-test on a phone. The goal isn't "a list of links" — it's letting them consume the material without ever leaving the task (timer running, draft safe). The design below is grounded in the actual authored content, not guesses.

Grounded in real prod content

source: prod-catalog-content.sql — the live tasks KMM & Sutirtha authored
🎧 Podcast — Spotify links ×35 ("Step 1: Listen to the podcast: open.spotify.com/…") 📄 Reading / PDF ×130 (self-hosted tasks/mini_timer/*.pdf, timed passages) 🎬 Video — YouTube ×~5 ("Muted Video Link: youtu.be/…") 🖼️ Passage image — Drive ×4 ("Picture: drive.google.com/…") 🔗 Web link ×few · only 4 real <a> tags · 0 iframes
The failure, precisely: material is buried as plain-text numbered steps inside the instructions (median ~321 chars). On a phone that's an untappable URL — and Spotify / YouTube / Drive throw the student out of the test (Drive can even hit a login/permission wall). That's why they "can't open" it. Drive is being retired — those 4 "Picture:" links convert to uploaded images.

One model → three surfaces, in sync

A single learning_materials list on the task is authored once and rendered three ways. Change it in authoring → the student's carousel and the evaluator's context both update. No drift possible.

  • 📄 Reading / PDF (self-hosted tasks/…pdf) → opens in an in-app viewer sheet; reuses the mini-timer passage the tasks already carry.
  • 🎬 Video (YouTube) → youtube-nocookie embed plays inline; no jump to the YouTube app.
  • 🖼️ Image → uploaded & self-hosted, opened in an in-app full-screen pinch-zoom viewer — exactly like the PDF. No external redirect.
  • 🔗 Web link → opens a new tab (the only true external), task/timer/draft preserved.
🚫 Google Drive links are not a supported type. Authoring rejects a Drive URL and asks for a PDF or image upload instead — so nothing depends on Drive sharing/permissions and nothing ejects the student.

The whole build — end-to-end & complete
1
Data model

New tasks.learning_materials JSON column: an ordered list of {type, title, url|path} where type ∈ podcast·video·pdf·image·link. One source of truth. Old task_multimedia kept & auto-folded in as item 1 (back-compat), then deprecated.

2
Authoring

The single Media picker becomes a drag-orderable repeater: pick type → paste URL (Spotify/YouTube/web) or upload (PDF/image, via UploadPolicy). Drive URLs are rejected inline — "upload the PDF/image instead". Gallery & mini-timer PDF untouched.

3
Student (mobile)

Swiper carousel of typed slides — Spotify/YouTube inline embeds, in-app PDF & image viewers, links new-tab — dots + "1/4" + peek + ✓-seen, plus the pinned Materials re-open chip on the answer screen. Reuses swiper taskCarousel.

4
Evaluator

CockpitPresenter gains materials (task-level, non-PII). The atelier shows a read-only "What the candidate was given" list beside the brief — open the exact podcast/passage to grade fidelity. Stays blind.

5
Data migration — tasks:extract-learning-materials

Idempotent, reversible, dry-run-first console command over every task:

  • Parse instructions/description + task_multimedia for URLs. Spotify → podcast, YouTube → video, self-hosted .pdf → pdf, other http(s) → link. Title = the step wording ("Listen to the podcast…").
  • Google Drive → NOT imported. Each Drive Picture:/file/d/… (the 4 today) is written to a conversion report (task id · url · surrounding text) for KMM to re-upload as an image/PDF. Where the Drive file is public & fetchable, an optional --rehost pass pulls it into our storage; anything gated is left in the report — never a broken slide.
  • Leaves the prose intact (materials become the structured layer; the step text stays as guidance) and records a per-task before/after so nothing is silently dropped.
6
Rollout & tests

Flag task_learning_materials_enabled — old blob renders until a task is migrated, so no flip-day risk. Tests: authoring (repeater save · Drive-reject · upload) · migration (each URL→type, Drive→report, idempotent, dry-run) · student render per type · evaluator context present + still blind. Then a real mobile browser E2E on one task of each media type before deploy.

Product call — how we deliver this

Choice was (a) a link inside the description vs (b) a "Learning Material" section. For a student testing on a phone:

B
"Learning Material" as a swipeable, type-aware carousel Recommended
Complete the existing shell into a Swiper deck of tailored slides (Spotify & YouTube inline · PDF/image full-screen · link new-tab) with dots + "1/4" + next-slide peek, plus a pinned Materials re-open chip on the answer screen. Auto-migrates the back-catalogue, retires the fragile raw-HTML blob (closes an XSS foot-gun).
A
Inline hyperlink in the description — fastest, but a lone link buried in prose is easy to skip, throws the student out to a browser tab mid-test, and can't carry a podcast/image/PDF. Fine as a secondary convenience, not the primary mechanism.

→ Proposed: build B as a carousel. Authoring gets an "Add learning material" repeater (type · title · URL or upload); the student task view renders the swipeable deck above + the pinned re-open chip. Casual inline links in the description keep working too.
